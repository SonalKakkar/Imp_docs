**3. How do you decide what to automate?**

I follow ROI and stability principles.

I prioritize:, High value candidates:, Regression scenarios, Stable features, Critical business flows, API tests (high ROI), Data validation scenarios, Avoid automating:, Frequently changing UI, One-time tests, Exploratory testing, I also use the automation pyramid:, 70% API tests, 20% integration, 10% UI

**4. How do you handle flaky tests?**

Flaky tests reduce trust in automation, so I treat them as defects.

My approach:, First identify root cause:, Usually:, Timing issues, Environment instability, Poor locators, Test data problems,

Then fix by: , Explicit waits instead of sleep, Stable selectors, Retry logic (limited), Data isolation

Process improvement:

I also maintain: Flaky test dashboard, Failure trend analysis, Quarantine unstable tests, My rule:, Automation must be trusted or it becomes useless.

**5. How do you integrate automation into CI/CD?**

My goal is continuous testing.



I integrate automation into CI/CD by embedding different test layers at appropriate stages of the pipeline.

Fast tests like unit and API run on every commit to provide quick feedback.

Heavier tests like integration and E2E run in later stages or in parallel to optimize execution time.

I use tagging and parallel execution to prevent pipeline slowdowns.

This ensures reliable quality checks without impacting delivery speed.
